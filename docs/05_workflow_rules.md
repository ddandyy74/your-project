## 需求管理

### 需求变更流程

1. 提出变更请求
2. 评估影响范围
3. PM 审批
4. 更新文档
5. 通知相关人

### 需求优先级

- P0: 阻塞问题，必须立即处理
- P1: 重要功能，本迭代完成
- P2: 期望功能，下迭代考虑
- P3: 锦上添花，有空再做

## 开发规范

### 代码审查

- PR 必须有 1 人以上 approval
- 代码审查24小时内响应
- 禁止自己合并自己的 PR

### 分支策略

```
main        # 生产分支
develop     # 开发分支
feature/*   # 功能分支
bugfix/*    # 修复分支
hotfix/*    # 紧急修复
```

### Commit Message 规范

```
<type>(<scope>): <subject>

Types:
- feat: 新功能
- fix: Bug 修复
- docs: 文档更新
- style: 代码格式
- refactor: 重构
- test: 测试
- chore: 构建/工具

示例:
feat(auth): 添加第三方登录
fix(payment): 修复支付超时问题
```

## 协作规则

### 每日站会

- 时间: 每天 10:00
- 时长: 15 分钟
- 内容: 昨日完成 / 今日计划 / 阻塞问题

### 迭代周期

- 长度: 2 周
- 计划会: 迭代第1天
- 评审会: 迭代最后1天
- 回顾会: 评审会后
